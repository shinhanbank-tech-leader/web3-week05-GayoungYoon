## 핵심 개념 정리
### EOA (Externally Owned Account)
이더리움의 기본 계정 형태로, ECDSA 개인 키로만 제어됩니다. 구조가 단순해 비용이 가장 적게 들지만, 단일 키 분실 = 자산 영구 손실이라는 근본적 취약점이 있습니다. 프로토콜을 변경하지 않는 한 이 구조는 바꿀 수 없습니다.
### ERC-4337 (Account Abstraction)
프로토콜을 건드리지 않고 스마트 컨트랙트 지갑을 구현한 표준입니다. 기존 tx 대신 UserOperation이라는 객체를 사용하고, EntryPoint 컨트랙트가 유일한 신뢰 지점으로 작동합니다.

### EOA vs ERC-4337 비교
| 구분 | EOA (Externally Owned Account) | ERC-4337 (Account Abstraction) |
|------|--------------------------------|--------------------------------|
| 제어 방식 | 개인 키 (ECDSA) | 스마트 컨트랙트 로직 |
| 스마트 컨트랙트 | 없음 | 있음 |
| 트랜잭션 서명 | 항상 개인 키 필요 | 커스텀 서명 가능 |
| 가스 지불 | 항상 ETH 보유 필요 | Paymaster 대납 가능 |
| 복구 방법 | 시드 구문만 | 소셜 / 다중 복구 |
| 다중 서명 | 기본 미지원 | 기본 지원 |
| 배포 비용 | 무료 | 가스 필요 |
| 표준 | 이더리움 기본 | EIP-4337 (2023) |

## EOA의 위험성과 불편함
EOA의 핵심 문제는 "개인 키 = 모든 것"이라는 구조에서 비롯됩니다. 키를 잃으면 자산도 사라지고, 키가 노출되면 즉시 전액 탈취가 가능합니다. 패스워드와 달리 리셋 수단이 프로토콜 레벨에 아예 존재하지 않습니다. 또한 매 트랜잭션마다 ETH를 가스비로 보유해야 하고, 여러 액션을 묶을 수 없으며, 다중 서명을 위해서는 별도 멀티시그 컨트랙트를 따로 배포해야 합니다.은행과 비교하면 EOA는 "OTP 시드 카드만 있고 영업점이 없는 계좌"와 같습니다. ERC-4337은 그 틈새를 스마트 컨트랙트 로직으로 채운 것으로, 은행이 제공하던 복구·위임·자동화 기능을 탈중앙 방식으로 재현하려는 시도입니다.

## ERC-4337 트랜잭션 흐름 및 핵심 기능
### 1. 트랜잭션 흐름
UserOperation (사용자 의도) → Bundler (묶음 전송) → EntryPoint (단일 컨트랙트) → Paymaster (가스 대납, 선택) → Smart Account (실행)

---
### 2. 핵심 기능
#### UX
##### Paymaster (가스 추상화)
- 제3자가 가스비 대납
- ERC-20 토큰으로 가스 납부 가능
- 또는 무료 온보딩 가능
---
#### 보안
##### 소셜 복구
- 신뢰하는 지인 또는 장치 기반 복구
- 키 분실 시 계정 복구 가능
##### 세션 키 / 권한 위임
- 특정 조건(금액, 용도, 기간)에 한해 동작하는 임시 키 발급
- 서비스 자동화 및 보안 강화
---
#### 확장성
##### 배치 트랜잭션
- approve + swap 등을 하나의 UserOperation으로 처리
- 단일 서명으로 여러 작업 실행
##### 커스텀 서명 검증
- Passkey, WebAuthn, BLS 등 다양한 인증 방식 지원
- 기존 개인 키 의존 구조 탈피
---
#### 자동화
##### 자동 실행 / 구독
- 조건부 트랜잭션 실행
- 정기 결제, 자동 리밸런싱 등 가능
---

## EOA vs 은행 계좌 vs ERC-4337 비교

| 항목 | EOA (개인 키 지갑) | 은행 계좌 (참고 비교) | ERC-4337 (Account Abstraction) |
|------|------------------|----------------------|--------------------------------|
| 계정 제어 방식 | 개인 키 (ECDSA) | 신원 기반 (KYC) | 스마트 컨트랙트 로직 |
| 복구 방식 | 불가능 (시드 구문만) | 신분 인증으로 복구 | 소셜 복구 / 다중 복구 |
| 가스/수수료 | 사용자 직접 부담 (ETH 필요) | 기업/기관 대납 가능 | Paymaster로 대납 가능 |
| 트랜잭션 처리 | 개별 실행 (각각 서명 필요) | 자동이체 / 일괄 처리 | 배치 트랜잭션 가능 |
| 자동화 | 불가능 | 정기결제 / 조건부 실행 | 자동 실행 / 구독 가능 |
| 권한 관리 | 없음 | 한도 / 권한 설정 가능 | 세션 키 / 권한 위임 |
| 보안 구조 | 단일 키 (SPOF) | 중앙 통제 + 인증 | 다중 서명 / 커스텀 인증 |
| 사용자 책임 | 100% 사용자 책임 | 기관이 일부 책임 | 사용자 + 시스템 분산 |
| 유연성 | 낮음 | 중간 | 매우 높음 |
| 개념 비유 | OTP 시드만 있는 계좌 | 관리자가 있는 계좌 | 규칙이 내장된 계좌 |


## 트릴레마 관점 비교 및 해결방향
### 트릴레마 기반 비교

| 트릴레마 축 | EOA | ERC-4337 | 핵심 트레이드오프 |
|------------|-----|----------|-------------------|
| 보안 | 단일 키 (SPOF) | 다중 서명 · 소셜 복구 | Bundler / Paymaster 신뢰 의존성 추가 |
| 분산화 | L1 직접 접근 | Bundler 중앙화 위험 | Bundler 시장 성숙도 의존, 검열 저항 과제 |
| 확장성 / UX | 매 tx 서명 · 가스 필요 | 배치 · 가스 스폰서 · 자동화 | 스마트 컨트랙트로 인한 추가 가스 비용 |
| 비용 | 가장 저렴 | 추가 overhead 발생 | EntryPoint 호출 가스 + Bundler 수수료 |


### 트릴레마와의 관계
블록체인 트릴레마(보안·분산화·확장성)는 지갑 레이어에서도 동일하게 나타납니다.
보안 향상, 분산화 희생: ERC-4337은 소셜 복구·다중 서명으로 보안은 강화되지만, Bundler가 UserOperation을 대신 제출하는 구조 때문에 Bundler 집중화와 검열 위험이 생깁니다. 현재 대부분의 트래픽이 소수의 Bundler에 의존하는 것이 현실입니다.
UX 향상, 비용 증가: Paymaster로 가스 추상화가 가능하지만 EntryPoint 호출 overhead로 순수 EOA보다 가스비가 높습니다.

### 트릴레마 해결 방향

EIP-7702 (Pectra 업그레이드 포함): EOA가 특정 tx에서 스마트 컨트랙트 코드를 임시 위임받을 수 있어, EOA를 유지하면서도 AA 기능을 활용합니다.
Native AA on L2: zkSync Era, StarkNet은 프로토콜 레벨에서 AA를 지원해 Bundler overhead를 없앴습니다. L1의 트릴레마를 L2로 분산하는 전략입니다.
탈중앙 Bundler mempool: Flashbots 등이 P2P Bundler 네트워크를 연구 중이며, 분산화 문제를 해결하려는 핵심 과제입니다.
Passkey 통합: WebAuthn 기반 생체 인증을 서명자로 삼아 시드 구문 자체를 없애는 방향으로, 일반 사용자 온보딩의 마찰을 획기적으로 줄이는 시도입니다.

## 블록체인 트릴레마와 L2의 등장
이더리움 L1은 보안과 분산화를 선택하고 확장성을 희생했습니다. 그 결과 트래픽이 몰리면 가스비가 폭등하고 처리량이 한계에 달합니다.L2가 트릴레마를 해결하기 위해 아래와 같은 3가지 방법을 사용합니다.L2는 실행을 오프체인에서 수행하고 결과 증명만 L1에 기록하는 방식으로 확장성을 확보합니다. Optimistic Rollup은 "일단 실행하고 이의 제기 기간(7일)을 두는" 방식이고, ZK Rollup은 수학적 증명으로 즉시 검증하는 방식입니다. 특히 ZK 계열 L2(zkSync, StarkNet)는 프로토콜 레벨에서 계정 추상화를 지원해, ERC-4337의 Bundler overhead 없이 AA 기능을 네이티브로 제공합니다.

- **Optimistic Rollup**
  - "일단 실행 → 문제 있으면 이의제기"
  - 구현 쉬움 but 출금 지연 존재

- **ZK Rollup**
  - "증명 후 즉시 검증"
  - 보안성과 확정성 우수

- **Native AA on L2**
  - 계정 추상화가 프로토콜에 내장
  - ERC-4337 구조 없이도 스마트 계정 사용 가능
  - Bundler / EntryPoint overhead 제거
 
## ERC-4337의 사용성 향상이 가져오는 새로운 통제 과제 + 금융회사 관점 현실적 구조
 
  사용성이 좋아질수록 기존 EOA에 없던 중간 행위자(Bundler, Paymaster)가 생기고, 복구 권한·승인 규칙을 누가 설계하느냐는 새로운 거버넌스 문제가 됩니다.

### ERC-4337 도입 시 통제 과제 및 금융회사 관점 정리

---

#### 1. 사용성 향상에 따른 새로운 통제 과제

| 구분 | 항목 | 내용 |
|------|------|------|
| 승인 구조 | 다중 서명 정책 | M-of-N 구조 설계, 임계값 변경 권한 정의 |
| 복구 권한 | 소셜 복구 모델 | Guardian 악의적 행위 대응, 내부자 공격, KYC 필요성 |
| 수수료 | Paymaster 부담 주체 | 가스비 대납 정책, 악용 방지, 한도/회계 처리 필요 |
| 장애 대응 | Bundler 장애 책임 | Bundler 다운 시 UserOp 전송 불가, SLA 및 대체 경로 필요 |
| 검열 저항 | Bundler 중앙화 | 특정 트랜잭션 차단 가능성, 금융 소비자 보호 이슈 |
| 컴플라이언스 | 로그 및 추적 | UserOperation 추적, AML/CTF, 회계 감사 대응 |

---

#### 2. 은행 vs 블록체인 (인증·승인·계정 관리 비교)

| 구분 | 은행 | 블록체인 |
|------|------|----------|
| 승인 방식 | DB 수정 가능 (중앙 통제) | 불변 (되돌릴 수 없음) |
| 인증 구조 | 다중 인증 (OTP, 생체 등) | 개인 키 또는 서명 |
| 권한 관리 | 한도, 계좌 제한 등 설정 가능 | 스마트 컨트랙트로 구현 필요 |
| 복구 방식 | 고객센터 / 신원 인증 | 기본적으로 불가능 (EOA) |
| 책임 구조 | 기관 책임 | 사용자 책임 (또는 분산 책임) |
| 중개자 역할 | 명확 (은행) | Paymaster / Bundler 등 새로운 중개자 |

---

#### 3. 금융회사 IT 관점: 현실적인 구조 (단계별)

| 단계 | 구조 | 설명 |
|------|------|------|
| 1 | Permissioned L2 + ERC-4337 | 컨소시엄 체인 위에서 AA 적용, Bundler 내재화, SLA 통제 가능 |
| 2 | EIP-7702 + 기존 EOA 병행 | 기존 사용자 유지하면서 점진적 AA 도입 |
| 3 | 내부 AA + 외부 L2 브리지 | 핵심 자산은 내부 체인, DeFi 연동은 외부 L2 활용 |
| 4 | Native AA L2 연동 | zk 기반 AA로 Bundler 제거, 장기적으로 확장 |


## 금융회사 IT 관점 종합 의견
개인적으로 가장 현실적인 접근은 "허가형 L2 위의 ERC-4337"이라고 생각합니다. 이유는 세 가지입니다.
첫째, 금융회사의 가장 큰 과제는 규제 대응과 감사 가능성입니다. 공개 메인넷에서 탈중앙 Bundler를 쓰면 특정 UserOp의 처리 여부를 통제할 수 없고 SLA 보장이 어렵습니다. 허가형 체인에서 Bundler를 내재화하면 이 문제가 해소됩니다.
둘째, Paymaster를 KYC 시스템과 연동하면 "허가된 고객에게만 가스 스폰서"라는 정책을 코드로 강제할 수 있습니다. 이는 기존 은행의 고객 등급별 수수료 면제와 동일한 논리를 온체인으로 구현한 것입니다.
셋째, EIP-7702가 안정화된 이후 EOA 지갑 사용자를 스마트 지갑으로 마이그레이션하는 경로가 열립니다. 당장 모든 걸 바꾸기보다 레거시 호환을 유지하면서 기능을 점층적으로 확장하는 방식이 현실적입니다.
단, 공개 DeFi 생태계와의 연동이 필요한 시점이 오면 결국 공개 L2와의 브리지 설계, 거기서 발생하는 자금 이동의 AML 추적 체계를 어떻게 구축하느냐가 중장기 핵심 과제가 됩니다.
